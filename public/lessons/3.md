### Theory

In the last lesson you saw that `README.md` and `app.js` are **untracked**. Git knows they exist, but it is not saving any changes to them. Before you can save a file, you need to tell git you want to include it. That is what `git add` does.

`git add` moves a file into the **staging area**. Think of the staging area as a box you are packing before sending a parcel. You choose exactly what goes in the box, then when you are happy you seal and send it with `git commit`.

**`git add <file>`**

Adds one specific file to the staging area.

**`git add .`**

The `.` means "everything in this folder". This stages all your files at once, which is handy when you want to save all your changes together.

When `git add` works, it says nothing. No output means success. Run `git status` after to see the file has moved.

### Further reading

**Unstaging a file**

Changed your mind? You can remove a file from the staging area with `git restore --staged <file>`. This puts it back in the working directory without changing its contents. Nothing is lost.

**Staging only part of a file**

In a real terminal you can run `git add -p` to stage just part of a file's changes. This tutorial does not support it, but it is worth knowing it exists for when you use git outside of here.

**`git add` and new vs modified files**

`git add` works for both brand-new files and files that have been modified since the last commit. In both cases you are telling git: "include the current state of this file in my next commit."

### Further reading button
Label: What does "tracked" mean in git?

Git divides files into two categories:

**Untracked** files are ones git has never been asked to watch. They exist on disk but git ignores them completely.

**Tracked** files are ones that have been added and committed at least once. Git watches them for changes and will tell you if something has been modified or deleted.

Running `git add` on an untracked file starts that tracking relationship. From the moment a file is first committed, git remembers every version of it for the lifetime of the repository.
