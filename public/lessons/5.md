### Theory

So far everything you have done has stayed on your own computer. `git push` is how you send your commits to a remote repository, such as one hosted on GitHub.

A **remote** is just another copy of your repository stored somewhere else, usually on the internet. When you push, git sends all the commits the remote doesn't have yet and brings it up to date.
===
**Setting up a remote**

Before you can push, git needs to know where to send your work. You register a remote with:

`git remote add origin <url>`

`origin` is just a name, but it is the universal convention for your main remote. The URL points to where the repository is hosted. Once you have added it, git remembers it.
===
**`git push`**

After adding a remote, `git push` sends your commits to it. You will see output showing the progress and a line like:

`main -> origin/main`

This means your local `main` branch has been pushed to `origin/main` on the remote. Both are now pointing at the same commit.

### Further reading

**What is GitHub?**

GitHub is a website that hosts git repositories. It is not part of git itself, it is just a popular place to store remotes. GitLab and Bitbucket are alternatives that work the same way.

When you push to GitHub, your code becomes accessible from any computer, and you can share it with collaborators.

**`git push` and branch tracking**

The first time you push a new branch, you may need to tell git where to push it with `git push -u origin <branch>`. The `-u` flag sets up a tracking relationship so that future `git push` and `git pull` commands on that branch know which remote branch to talk to.

**What about `git pull`?**

`git pull` does the opposite of `git push`. It downloads commits from the remote that you don't have locally yet. You will use it whenever a colleague has pushed new work and you need to get their changes. We cover this in a later lesson.

### Further reading button
Label: How does GitHub fit in?

When you create a repository on GitHub and add it as a remote, you are connecting your local git history to a copy stored on GitHub's servers.

Every time you `git push`, GitHub receives your new commits and stores them. Your repository page on GitHub always shows the latest pushed state of your code.

This is why GitHub is so useful for collaboration. Two developers can both clone the same repository, work independently, and push their changes. GitHub holds the shared history that everyone contributes to and pulls from.
