### Theory

Before git can do anything useful, you need to tell it to start watching your project. You do that with one command: `git init`.

Think of it like pressing record on a camera. Nothing is being captured yet — you're just switching it on.

Running `git init` sets up git in your current folder. You only ever do this once per project.
===
**`git status`**

`git status` is your way of asking git "what's going on right now?" It shows you which files have changed, which are ready to save, and which git hasn't seen before.

It never changes anything — it's always safe to run, and you'll use it constantly.

Run `git status` to see the current state of your repository.

You should see two things: the branch you are on (`main`), and a list of **untracked files** — `README.md` and `app.js`. These files exist on disk but git is not tracking them yet. The next lesson covers how to change that.

### Further reading

**What does "Branch" mean?**

When git says `On branch main`, it means you are working on a named line of development called `main`. You can think of it as the primary timeline of your project. Later lessons cover creating and switching between branches.

**Why does git need `init`?**

Not every folder on your computer is a git repository. Running `git init` is an explicit choice to start tracking a project. This means you can have regular folders sitting right next to git repositories and they will not interfere with each other.

**The `.git` folder**

You will rarely need to look inside `.git` directly, but it is worth knowing it exists. It contains the full history of your project, all branches, and git's configuration. Deleting it would remove all version history (but leave your working files intact).

### Further reading button
Label: What is version control?

Version control systems like git solve a fundamental problem in software development: how do you safely make changes to a codebase without losing work or overwriting a colleague's changes?

**Before version control**, teams would email zip files, manually merge changes, and keep folders full of dated copies. This was error-prone and did not scale.

**With git**, every change is recorded with a message, an author, and a timestamp. You can see the full history of any file, revert a bad change in seconds, and work on experimental features in isolation without touching the main codebase.

Git was created by Linus Torvalds in 2005 to manage the Linux kernel — one of the largest open-source projects in the world. Today it is the standard version control tool across the industry.
