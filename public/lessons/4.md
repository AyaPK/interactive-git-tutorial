### Theory

You have staged your files. Now it is time to save them permanently. That is what `git commit` does.

A commit is a snapshot of everything in your staging area at that moment. Git stores it forever (or until you deliberately delete it). Every commit gets a unique ID, an author, a timestamp, and a message you write yourself.

Think of it like saving a document, but instead of overwriting the previous version, git keeps every save you have ever made. You can always go back.

**`git commit -m "your message"`**

The `-m` flag lets you write your commit message inline. The message should be short and describe what changed.

```
git commit -m "add README and app files"
```

A good commit message finishes the sentence: "If applied, this commit will..." For example: "add login page", "fix typo in README", "remove unused variable".

**What happens after a commit**

The staging area is cleared. Your files are safely stored in git's history. Run `git status` and you will see "nothing to commit, working tree clean".

### Further reading

**Why write good commit messages?**

When you look back at a project months later, commit messages are how you understand what changed and why. "fixed stuff" tells you nothing. "fix crash when user submits empty form" tells you exactly what happened.

A common convention is to write messages in the imperative tense, as if giving an instruction: "add feature", not "added feature" or "adding feature". Many teams enforce this as a standard.

**What is a commit hash?**

Every commit gets a long string of letters and numbers called a hash, like `a3f2c91`. This uniquely identifies that snapshot. You will see a shortened version of it in the commit output. You can use hashes to reference specific commits in other git commands.

**Amending a commit**

Made a typo in your last commit message, or forgot to include a file? You can fix the most recent commit with `git commit --amend`. This rewrites the last commit rather than creating a new one. Only do this before you have shared the commit with others.

### Further reading button
Label: What makes a good commit?

A good commit does one thing and describes it clearly.

**Small and focused** commits are easier to review, easier to revert if something goes wrong, and easier to understand later. A commit that changes 500 lines across 20 files is very hard to reason about.

**A common approach** is to commit every time you finish a small, self-contained piece of work. Fixed a bug? Commit. Added a new function? Commit. Updated some documentation? Commit.

Over time your commit history becomes a readable diary of how your project evolved, which is one of the most powerful things git gives you.
